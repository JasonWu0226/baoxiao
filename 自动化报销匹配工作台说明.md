# 自动化报销匹配工作台说明

## 当前版本目标

新程序从 `报销准备的资料` 里的原始素材开始整理，不再要求先在 Excel 里确认。发票下载、发票有效性核验、订单/付款截图确认、发票与付款匹配，都尽量在软件内完成。

核心开发原则是：日常报销只能依赖运行软件完成，不能依赖每次回到 Codex 里临时处理。Codex 只用于把新问题开发成软件能力；凡是当前需要手动跑脚本、手动搬文件、手动查漏、手动判断的动作，都应该逐步变成软件里的入口、确认记录和可重复执行流程。

所有人工确认动作会保存到：

- `规则/确认记录.json`
- `规则/判断权重.json`

后续再次扫描时，软件会复用这些记录，逐步提高自动判断的准确度。

## 基本流程

1. 双击 `运行自动化报销匹配工作台.bat`。
2. 检查顶部的素材目录、发票目录、模板、输出目录和日期范围。
3. 如果有上期材料，先放到 `报销准备的资料/上一期报销`，或在顶部 `上期发票目录` 填入路径。
4. 在 `下载完整性核验` 或 `邮件发票判断` 页面点击 `更新上期报销索引`，生成 `规则/上期报销索引.json` 和 `输出/上期报销查询表_时间.xlsx`。
5. 在 `发票下载核验` 页面配置邮箱。
6. 输入邮箱授权码，点击 `保存配置` 后再点击 `下载邮箱发票`。
7. 下载时软件会先查上期索引，命中上期的邮件标记为 `疑似上期已报销跳过`，不重复下载。
8. 下载完成后，在 `邮件发票判断` 和 `下载完整性核验` 中处理剩余待确认项。
9. 在 `逐项确认` 中确认淘宝、微信、美团、出行等付款/订单素材是否报销。
10. 在 `发票/付款匹配` 中确认系统生成的候选匹配。
11. 点击 `保存确认记录`。
12. 点击 `生成报销Excel`。

## 上期报销索引

为了避免本期重复处理上期已经报销过的发票，软件会先基于上期材料生成查询索引：

- 自动扫描 `上期发票目录`，以及 `报销准备的资料/上一期报销`、`报销准备的资料/上期报销`、`报销准备的资料/历史报销` 等目录。
- 读取上期 PDF 发票的发票号、金额、开票日期、销售方、文件哈希和文件路径。
- 记录上期报销 Excel、Word 等报销文档路径。
- 保存 JSON 到 `规则/上期报销索引.json`，供软件自动判断使用。
- 生成 Excel 到 `输出/上期报销查询表_时间.xlsx`，供人工查询和核对。

邮箱下载前，软件会用邮件主题、发件人、附件名、取票链接里的发票号、金额和销售方信息查询这个索引。高置信命中的邮件会标记为 `疑似上期已报销`，不会进入普通下载和本期报销流程。它不是“无发票”，而是“有发票但疑似上期已经处理过”。

## 邮箱发票下载

当前支持通过 IMAP 下载邮箱里的发票：

- 默认服务器：`imap.qq.com`
- 默认端口：`993`
- 默认文件夹：`inbox`
- 默认下载目录：`所有发票`
- 支持按开始日期和结束日期筛选邮件
- 支持下载邮件附件中的 PDF/OFD/XML/图片/压缩包
- 支持从邮件正文里的发票下载链接继续下载文件
- 如果正文链接打开后是取票网页，软件会解析网页里的 PDF/OFD/XML/ZIP 候选下载地址并继续尝试下载。

QQ 邮箱通常需要使用 IMAP/SMTP 授权码，不是网页登录密码。授权码会随 `保存配置` 写入 `matcher_config.json`，下次打开软件会自动带出，不需要重复输入。这个配置文件保存在本机，属于明文保存，建议只在个人电脑上使用。

当前 C# 工作台已经并入更严格的自动化发票邮件下载与管理逻辑：

- 每封邮件都会生成邮件级处理记录，包含邮件 ID、主题、发件人、接收时间、是否含附件、是否为发票邮件、处理状态、下载状态、异常原因。
- 只有邮件标题或正文命中 `发票/电子发票/增值税发票/专票/普票/数电票/报销/开票/invoice` 等关键词时，才会进入发票下载流程。
- 同一封邮件同时存在 PDF 和图片/OFD/XML/ZIP 时，优先只处理 PDF；没有 PDF 时才保留其他合法格式，并在邮件状态中标记 `异常 / 无 PDF`，方便人工复核。
- 下载文件会按 `发票_YYYYMMDD_发件人_唯一标识` 的规则命名。
- PDF 下载后会尝试提取发票号、发票金额、开票日期；发票号无法识别时标记 `异常 / 无发票号`。
- 查重依据包含文件 MD5/SHA256、发票号码、发票金额、开票日期；重复发票会标记 `重复发票 / 重复已忽略`，不重复保存文件。
- 如果 PDF 内识别出的开票日期早于下载开始日期，会标记 `异常 / 发票日期早于开始时间`。
- 发票日期过滤以发票 PDF/XML 内识别出的开票日期为准，不以邮件接收日期为准。邮件是 2026 年收到，但发票开票日期是 2025 年 12 月的，会按日期不符处理。

每次下载后会在发票目录生成：

- `邮箱发票下载清单_yyyyMMdd_HHmmss.csv`

这份清单用于核对哪些邮件、附件或链接被处理过。真正的确认动作仍然在软件的 `发票下载核验` 页面里完成。

软件里还有 `下载完整性核验` 页面，用来查看最新下载清单：

- `已下载`：文件已经正常保存。
- `已处理跳过`：该邮件此前已经成功处理且无异常，本次不重复下载。
- `无发票内容已跳过`：该邮件没有识别到发票附件或疑似发票链接，后续不重复处理。
- `重复跳过` / `重复已存在`：同一文件之前已经下载过，本次不再生成 `_1` 副本。
- `PDF已存在跳过`：同一张发票已经有 PDF，OFD/XML/图片等非 PDF 格式不再下载。
- `未下载到文件`：该邮件在当前规则下没有保存出发票文件，需要人工核验。
- `失败`：附件或链接下载失败。
- `需人工确认`：邮件里有疑似发票链接，但链接不是可直接下载的 PDF/OFD/XML/ZIP，需要人工打开核对。
- `正常已下载`：已经保存有效 PDF 发票。
- `重复发票`：与历史已下载发票重复，已经忽略重复文件。
- `异常`：命中发票邮件但存在无附件、无 PDF、无发票号、下载失败、发票日期早于开始时间等问题。
- `非发票邮件`：没有命中发票关键词，记录后跳过。
- `页面解析已下载`：取票链接本身不是文件，但软件从网页里解析到了真正的发票下载地址并保存成功。
- `链接取票待处理`：取票页面需要验证码、登录、扫码，或页面里没有可直接下载的文件地址，需要人工打开链接处理。
- `附件已取得发票，链接跳过`：附件中已经取得发票，正文里的取票链接不再处理。
- `发票日期早于开始时间跳过`：发票本身的开票日期早于邮箱下载开始日期，软件会归档该文件，不纳入本期下载成功和报销确认。

后续重新执行邮箱下载时，软件会记录每封邮件的附件数量、疑似发票链接数量、实际下载数量，方便确认是否所有发票都已经下载下来。

邮件处理状态会保存到 `规则/邮箱处理记录.json`。后续重复执行下载时，已经成功处理且无异常的邮件会自动跳过，只保留失败、需人工确认、未下载到文件的邮件继续提醒。

`下载完整性核验` 页面新增 `只重处理异常邮件`，用于只重新处理 `异常/失败/未下载到文件/需人工确认` 的邮件，不会重新跑已经正常完成、重复发票或非发票邮件。

`下载完整性核验` 页面也可以点击 `打开选中链接/文件`。如果选中行已有本地文件，会打开文件；如果没有文件但有链接，会用浏览器打开取票链接，适合处理验证码、登录、扫码等无法全自动下载的页面。

各个列表默认勾选 `只看待处理/异常`，已经确认有效、已排除、已匹配或已处理跳过的记录不会重复打扰。取消勾选后可以查看完整记录。

新增 `邮件发票判断` 页面，用来按“邮件”而不是按“附件行”确认发票下载规则：

- `明确有发票`：程序已经下载到文件，或附件/链接明显是发票。
- `明确无发票`：没有发票关键字、没有平台关键字、没有疑似发票附件或链接。
- `需要人工确认`：邮件可能有发票，但规则还不能稳定判断，需要人工看邮件主题、附件、链接和下载状态。

明确有发票和明确无发票的邮件不会反复拉出来确认；只有 `需要人工确认` 的邮件会保留在默认列表里。人工确认结果保存到 `规则/邮件判断记录.json`，点击 `保存下载规则` 会同步生成 `规则/邮件下载规则.json`。后续重新执行邮箱下载时，人工确认为 `明确无发票` 的邮件会直接跳过，不再重复下载和核验。

`邮件发票判断` 页面也可以生成 `输出/邮件发票判断清单_时间.xlsx`，表里包含 `需要人工确认`、`全部邮件` 和 `统计汇总` 三个工作表，方便你整体检查当前邮箱发票是否已经覆盖。

`下载完整性核验` 页面可以点击 `生成邮件统计Excel`，输出 `输出/邮箱逐封统计_时间.xlsx`。这份表会逐封邮件统计：

- 是否命中发票关键字
- 是否命中美团、淘宝、京东关键字
- 是否有附件
- 附件是否疑似发票
- 是否有疑似发票链接
- 下载文件数、重复/跳过数
- 是否需要人工核验
- 是否建议模型判断

模型判断建议只针对规则无法确定的邮件，不建议每封邮件都调用模型。当前版本已经支持 OpenAI-compatible 大模型接口：

- 在 `发票下载核验` 页面填写 `大模型辅助判断` 配置。
- 小米 MiMo 可使用 `BaseUrl = https://api.mimo-v2.com/v1`。
- 文本判断可用 `mimo-v2.5-pro`，图片判断建议使用 `mimo-v2-omni`。
- 点击 `测试大模型` 可以检查 API Key、模型名和网络是否正常。
- 在 `邮件发票判断` 页面点击 `大模型判断待确认`，软件会把当前规则无法确定的邮件交给模型辅助判断，并把结果写入 `规则/邮件判断记录.json`。
- 模型置信度低于阈值时仍保留为 `需要人工确认`，不会强行标记有/无发票。

如果历史目录里已经存在同一张发票的 PDF/OFD/XML 多种格式，可以在 `下载完整性核验` 页面点击 `归档非PDF重复格式`。软件会把同一发票已有 PDF 的非 PDF 文件移动到 `输出/归档/非PDF重复格式_时间`，不直接硬删除。

如果当前发票目录里混入了上期或早于开始日期的 PDF，可以点击 `归档日期不符发票`。软件会读取 PDF 内的开票日期，把早于 `邮箱开始日期` 的 PDF 移动到 `输出/归档/日期不符发票_时间`，后续扫描、查重和报销生成都会跳过这些归档文件。

发票下载完成后，可以在 `下载完整性核验` 页面点击 `归集PDF发票并汇总`。软件会递归扫描发票目录里的 PDF，按发票号优先、文件哈希兜底去重，然后复制唯一 PDF 到 `输出/PDF发票归集_时间/PDF发票`，并生成 `PDF发票归集核验.xlsx`：

- `唯一PDF发票`：所有去重后的 PDF 发票清单和归集后路径。
- `待补充核验`：金额、发票号、开票日期或类别未识别的 PDF。
- `识别问题诊断`：只列真正的识别问题，并给出原因和建议动作。例如滴滴行程单会标记为 `非发票PDF`，因为它是出行明细，不包含发票号码。
- `上期重叠核验`：与上期已报销发票目录命中的发票，优先按发票号比对，其次按文件哈希，再按日期+金额+销售方做疑似比对。
- `重复PDF`：被去重跳过的 PDF 以及对应保留文件。
- `合并报销建议`：目前会把高速/通行费、滴滴/网约车按月份汇总，方便后续作为合并报销项处理。
- `总览`：唯一 PDF 数量、重复数量、待核验数量和分类金额汇总。

如果需要和上期报销对齐，在顶部 `上期发票目录(分号分隔)` 填入上期已经报销过的 PDF 目录，例如：

`早期资料/合并发票;早期资料/1-4月发票`

也可以直接把上期材料放到 `报销准备的资料/上一期报销`、`报销准备的资料/上期报销`、`报销准备的资料/历史报销` 等目录下，软件会自动把这些目录里的 PDF 发票纳入上期核验。填好或放好后再点击 `归集PDF发票并汇总`。命中的重叠项不会自动删除，会在表里标记 `疑似上期已报销`，由你确认是否排除本期报销。

对于非 PDF 图片和二维码，可以在 `下载完整性核验` 页面点击 `处理非PDF/二维码/AI判断`：

- `.ico` 图标会直接归档，因为它不可能是发票。
- 图片文件会先尝试识别二维码；如果二维码里是取票链接，软件会尝试自动下载 PDF/OFD/XML/ZIP。
- 如果没有二维码且已经启用大模型，软件会把图片交给视觉模型判断是否为发票图片。
- 模型判断为无关图片且置信度达到阈值时会归档；判断为发票图片时保留；不确定时继续留给人工确认。

扫描发票目录时，软件会自动处理历史 ZIP 压缩包：

- ZIP 里的 PDF/OFD/XML 会解压到 `所有发票/_zip解压缓存`。
- 如果 ZIP 里有 PDF，只保留 PDF 进入识别流程。
- 外层 ZIP 本身不会作为发票参与匹配，避免金额和发票号识别为 0 或空。
- 同一个 ZIP 重复出现时，按文件哈希复用同一个解压缓存。

## 自动化记录

- `规则/确认记录.json`：保存每一次素材确认和匹配确认。
- `规则/判断权重.json`：根据历史确认记录汇总来源、商户、事项的倾向。
- Excel 中绿色表示已确认，黄色表示待确认或建议项。

## 当前能力边界

当前版本已经打通软件内确认闭环和邮箱发票下载闭环。PDF/OCR 深度解析仍然是下一步增强点：

- 淘宝 PDF、微信交易明细 PDF、美团截图现在先作为原始证据进入确认工作台。
- 后续可以继续接入 OCR/PDF 解析器，把 PDF 和长截图拆成更细的订单或流水行。
- 金额不一致、日期不一致、订单多发票或发票多订单的情况，先进入候选匹配页人工确认；确认结果会沉淀为后续自动判断权重。
- 发票金额和发票号码会优先读取电子发票 XML；没有 XML 时会尝试读取 PDF 文本；仍无法识别的项目会在“建议”列标记为 `金额未识别` 或 `发票号未识别`。
